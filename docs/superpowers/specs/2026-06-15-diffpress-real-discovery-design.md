# DiffPress — Real Discovery & Drafting Columns (de-mock the board)

**Date:** 2026-06-15
**Status:** Approved-in-principle; one open decision flagged (discovery model A vs B)
**Related:** `2026-06-14-diffpress-content-engine-design.md`,
`2026-06-15-diffpress-ui-integration-design.md`

## Problem

The `/diffpress` pipeline board has four columns. Two are backed by the real
`PublicationLifecycle` ledger (**Ready-for-Dev** = `AWAITING_HANDOFF`,
**In-Review** = `PUBLISHED`). The other two — **Discovery** and **Drafting** —
are decorative mock data hardcoded in `src/components/diffpress/data.ts`
(`PIPELINE.discovery`, `PIPELINE.drafting`), merged in via `structuredClone()`
in `services.ts`.

A stale stub-era row also sits in the deployed ledger: a `PUBLISHED` record for
`pewdiepie-archdaemon/odysseus` ("A Critical Read of …"), generated by the old
`draftArticle` stub before tonight's Gemini drafter shipped.

## Core architectural principle (corrected)

**The UI never talks to GitHub.** All four columns are driven entirely by the
`PublicationLifecycle` table state, read through the existing
`GET /api/handoffs`. Discovery is an **asynchronous background process** (the
EventBridge-triggered Step Function), not a live UI fetch. This keeps every card
the UI shows a real pipeline citizen rather than an un-enriched GitHub result
that bypasses enrichment / idea-matching / handoff generation.

A repo's lifecycle becomes four persisted statuses:

```
DISCOVERED  ──►  AWAITING_HANDOFF  ──resume──►  DRAFTING  ──►  PUBLISHED
(Discovery)      (Ready-for-Dev)                (Drafting)     (In-Review)
```

## Goals

1. Add `DISCOVERED` and `DRAFTING` to the `PublicationStatus` union.
2. `listHandoffs.ts` buckets the board into **four** arrays driven entirely by
   DynamoDB state — no GitHub call in the read path.
3. Discovery candidates are written to the ledger by the background Step
   Function; **deduplication uses `BatchGetItem`, never a table scan**.
4. Drafting is a real `DRAFTING` status written by `draftArticle`.
5. Remove the pipeline-board mock from `data.ts` / `services.ts`.
6. Delete stale test rows from the deployed table.

## Non-Goals

- Touching the **Draft editor / Marginalia / deploy console** demo
  (`ARTICLE_HTML`, `TECH_EDITOR_NOTES`, `EMPTY_DEPLOY`). No backend, separate
  deferred feature — stays.
- Star-deltas ("+N this week") or coverage scores — both need N+1 external calls.
  Discovery cards show only what one GitHub search returns.
- Replacing the board-read `Scan` with a status GSI (see Open Questions).

## OPEN DECISION — discovery model (pick one before implementation)

The Step Function processes one repo per execution and pauses once
(`waitForTaskToken`). To populate `DISCOVERED` **and** `AWAITING_HANDOFF` as two
distinct columns, choose:

- **Model A (this spec's default):** `DiscoverRepos` writes **all top-N deduped
  candidates** as `DISCOVERED` (cheap metadata, no enrichment), then enriches +
  hands off the one it selects, which `notifyHandoff` flips to
  `AWAITING_HANDOFF`. Discovery column = un-selected candidate pool. No new UI
  action. `DISCOVERED` rows carry a DynamoDB **TTL** (~30d) so the pool
  self-expires without a cleanup scan.
- **Model B:** only the enriched, selected repo is written `DISCOVERED` at the
  pause; reaching `AWAITING_HANDOFF` requires a **new `POST /api/claim-handoff`**
  action (UI button + endpoint). Discovery shows ~one card per run.

Everything below is written for **Model A**; switching to B changes §1 and adds
a claim endpoint.

## Design

### 1. Discovery — written by the background Step Function (Model A)

In `discoverRepos.ts` (the EventBridge-triggered `DiscoverRepos` Lambda):

1. Run the existing GitHub search (`created:>30d, stars:>50`, top ~50 by stars).
2. **Dedup via `BatchGetItem`**, not a scan: issue a single `BatchGetItem`
   against `PublicationLifecycle` with the candidate `repoName`s as keys
   (partition key). Drop any candidate that returns an item — it is already
   somewhere in the pipeline (any status). 50 keys fit one `BatchGetItem` call
   (limit 100). Add a pure `dedupeByBatchGet(candidates, existingNames)` helper
   for unit-testability; the `filterUnpublished` scan-based path is removed.
3. **Write the surviving candidates as `DISCOVERED`** rows (PK `repoName`) with:
   `status: "DISCOVERED"`, `repoUrl`, `description`, `stars`, `language`,
   `pushedAt` (GitHub `pushed_at`), `discoveredAt`, and a numeric `ttl`
   (`now + 30d`, epoch seconds; enable TTL on the table for this attribute).
   Use `BatchWriteItem` (≤25/batch) for the writes.
4. Select one candidate and continue the existing flow (Enrich → SeedIdeas →
   `notifyHandoff`). `notifyHandoff` updates the **selected** repo's row from
   `DISCOVERED → AWAITING_HANDOFF`, attaching `taskToken` + `payloadKey`.

No `GET /api/discovery` endpoint. No `GITHUB_TOKEN` exposure to the API/UI.

**Discovery card fields (real GitHub data only):** repo name (`full_name`),
description, absolute `stars` (`stargazers_count`), `language`, last-updated
(`pushed_at`). The fabricated `starsDelta` / `coverage` fields are removed from
the frontend `DiscoveryCard` type.

### 2. Drafting — `DRAFTING` status, written by `draftArticle` (Approach A)

- Add `DRAFTING` to `PublicationStatus`.
- `draftArticle.ts` flips the row `→ DRAFTING` at handler start (before the
  Gemini call) via a `markDrafting(repoName)` ledger helper + pure
  `buildMarkDraftingParams(table, repoName)` (mirrors `buildMarkPublishedParams`).
  Link `PublicationLifecycle` to the `draftArticle` function for write IAM.
- Existing `buildMarkPublishedParams` condition
  (`attribute_not_exists(#status) OR #status <> :published`) already permits
  `DRAFTING → PUBLISHED` unchanged.

### 3. `listHandoffs.ts` — four-column bucketing, DynamoDB-driven

- `bucketBoard` maps every item by status into four arrays:
  `DISCOVERED → discovered`, `AWAITING_HANDOFF → readyForDev`,
  `DRAFTING → drafting`, `PUBLISHED → inReview`.
- `Board` interface: `{ discovered, readyForDev, drafting, inReview }`.
- `listBoardItems` projection extends to include the discovery fields
  (`stars, language, pushedAt, description`) so discovery cards render.
- The read path makes **no GitHub call**.

### 4. Frontend rewire (`services.ts`, `store.ts`, `types.ts`)

- `fetchCandidates()` maps **all four** columns from the single `/api/handoffs`
  response. Both `structuredClone(PIPELINE.*)` calls are deleted.
- Store initial state (`store.ts:128`) seeds empty columns instead of
  `structuredClone(PIPELINE)`.
- `DiscoveryCard` / `DraftingCard` frontend types updated to the real shapes.

### 5. Cleanup

- Delete `PIPELINE`, `HANDOFFS`, `REVIEW_BLOCKS` from `data.ts`. Keep
  `ARTICLE_HTML`, `TECH_EDITOR_NOTES`, `EMPTY_DEPLOY`. Remove dangling imports.
- Delete stale ledger rows (`pewdiepie-archdaemon/odysseus` + any other test
  rows) from the deployed `PublicationLifecycle` table via a one-off
  `delete-item` — a confirmed manual data step in the plan, not code.

## Data flow (after)

```
EventBridge ─► DiscoverRepos ─► GitHub Search ─► BatchGetItem dedup
                  │                                  │
                  ▼ (write survivors)                ▼ (drop existing)
            PublicationLifecycle  ◄── status: DISCOVERED (ttl 30d)
                  │  select one → Enrich → SeedIdeas → notifyHandoff
                  ▼  DISCOVERED → AWAITING_HANDOFF (+taskToken)
                  … resume → DRAFTING → PUBLISHED

UI ── GET /api/handoffs ─► listBoardItems (Scan) ─► bucketBoard →
        { discovered, readyForDev, drafting, inReview }   (no GitHub call)
```

## Testing

- **Pure units (vitest, no AWS):**
  - `dedupeByBatchGet` — drops candidates whose `repoName` is in the existing set.
  - `bucketBoard` — asserts all four statuses route to the right arrays.
  - `buildMarkDraftingParams` — asserts UpdateCommand shape.
  - Discovery candidate → ledger-row mapping (fields + `ttl` computed).
- **Handlers:** `discoverRepos` — mock Octokit + DynamoDB, assert `BatchGetItem`
  dedup, `BatchWriteItem` of `DISCOVERED` rows, single selection continues.
  `draftArticle` — assert `markDrafting` called at start.
- Typecheck + full vitest suite green + production build OK before merge.
- **Manual E2E:** trigger discovery on a real demo repo with `DIFFPRESS.md`;
  confirm a card appears in Discovery, advances to Ready-for-Dev on selection,
  to Drafting on resume, and to In-Review with a genuine Gemini headline.

## Open Questions (for the plan, non-blocking except the first)

1. **Model A vs B** above — must be settled before implementation.
2. **Board read still `Scan`s** `listBoardItems` (pre-existing). At scale a
   `status-index` GSI would replace it with per-status `Query`s. Recommended
   follow-up; out of scope here unless you want it folded in now.
3. Discovery `BatchWrite` count / TTL window (default top-N to write, 30d) —
   tune in the plan.
