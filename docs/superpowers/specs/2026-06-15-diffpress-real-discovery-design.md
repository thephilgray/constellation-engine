# DiffPress — Real Discovery & Drafting Columns (de-mock the board)

**Date:** 2026-06-15
**Status:** Approved design, ready for implementation plan
**Related:** `2026-06-14-diffpress-content-engine-design.md`,
`2026-06-15-diffpress-ui-integration-design.md`

## Problem

The `/diffpress` pipeline board has four columns. Two are backed by the real
`PublicationLifecycle` ledger (**Ready-for-Dev** = `AWAITING_HANDOFF`,
**In-Review** = `PUBLISHED`). The other two — **Discovery** and **Drafting** —
are decorative mock data hardcoded in `src/components/diffpress/data.ts`
(`PIPELINE.discovery`, `PIPELINE.drafting`), merged in via
`structuredClone()` in `services.ts`.

The original content-engine spec deliberately deferred these as mock. The
real `discoverRepos` Lambda already runs a genuine GitHub search but throws
away every candidate except the one it threads into the workflow; nothing
persists or exposes the rest. And there is no durable "currently drafting"
state, because drafting runs server-side automatically right after the human
handoff resumes (`AwaitHandoff → DraftArticle → RecordPublication`).

A stale stub-era row also sits in the deployed ledger: a `PUBLISHED` record
for `pewdiepie-archdaemon/odysseus` (headline "A Critical Read of …"),
generated by the old `draftArticle` stub before tonight's Gemini drafter
shipped.

## Goals

1. **Discovery column = real data**, sourced from a live GitHub query, not mock.
2. **Drafting column = real data**, sourced from a new `DRAFTING` ledger status.
3. **Remove the pipeline-board mock** from `data.ts` and `services.ts`.
4. **Delete stale test rows** from the deployed `PublicationLifecycle` table.

## Non-Goals

- Touching the **Draft editor / Marginalia / deploy console** demo. Those use
  `ARTICLE_HTML`, `TECH_EDITOR_NOTES`, and `EMPTY_DEPLOY` in `data.ts`, have no
  backend, and are a separate deferred feature. They stay.
- Computing star-deltas ("+N this week") or coverage scores. Both require N+1
  external calls (OSSInsight/historical store for deltas; Exa/Tavily per repo
  for coverage) — exactly the cost the funnel exists to avoid. Discovery cards
  show only what one GitHub search returns.
- A real-time SSE/websocket board. The board refreshes on load/poll as today.

## Design

### Lifecycle

```
DISCOVERED (ephemeral, live GitHub — never persisted)
        │  selected by discoverRepos + human handoff
        ▼
AWAITING_HANDOFF  ──resume──►  DRAFTING  ──recordPublication──►  PUBLISHED
   (Ready-for-Dev)              (Drafting)                       (In-Review)
```

Discovery is intentionally **not** a ledger status — it is a live read. Only
`AWAITING_HANDOFF → DRAFTING → PUBLISHED` are persisted.

### 1. Discovery — live, isolated endpoint

**New handler `src/diffpress/listDiscovery.ts` → `GET /api/discovery`** (behind
the existing Cognito JWT authorizer; `GITHUB_TOKEN` linked).

- Refactor the GitHub-search query out of `discoverRepos.ts` into a shared
  helper (e.g. `lib/discovery.ts`) exporting `searchEmergingRepos()` returning
  `RepoCandidate[]`, and reuse the existing `filterUnpublished`. Both the
  Step Functions Lambda and the endpoint call the same path — one source of
  truth for "what counts as an emerging repo."
- The endpoint filters candidates against **all** ledger repoNames (any status),
  not just `PUBLISHED`, so a repo already in the pipeline never double-appears
  in Discovery. Add a `listAllRepoNames()` ledger helper (scan, project
  `repoName`) — or extend `listPublishedNames` usage; implementation plan to
  pick the lighter option.
- Returns the top ~6–8 candidates.
- **Failure isolation:** kept separate from `/api/handoffs` on purpose. A GitHub
  outage degrades only the Discovery column; the ledger-backed columns still
  load.

**Discovery card fields (real GitHub data only):**

| Card field      | GitHub source            |
|-----------------|--------------------------|
| repo name       | `full_name`              |
| description     | `description`            |
| stars (absolute)| `stargazers_count`       |
| language        | `language`               |
| last updated    | `pushed_at`              |

The fabricated `starsDelta` and `coverage` fields are removed from the
frontend `DiscoveryCard` type.

### 2. Drafting — `DRAFTING` ledger status (write owned by `draftArticle`)

- Add `DRAFTING` to the `PublicationStatus` union in `src/diffpress/types.ts`.
- **`draftArticle.ts` flips the row to `DRAFTING` at handler start**, before the
  Gemini call, via a new ledger helper `markDrafting(repoName)` with a pure
  param-builder `buildMarkDraftingParams(table, repoName)` (mirrors the existing
  `buildMarkPublishedParams`). Link `PublicationLifecycle` to the `draftArticle`
  function for write IAM.
- The existing `buildMarkPublishedParams` condition
  (`attribute_not_exists(#status) OR #status <> :published`) already permits a
  `DRAFTING → PUBLISHED` transition unchanged.
- `bucketBoard` (in `listHandoffs.ts`) grows a third bucket `drafting`; the
  `Board` interface gains `drafting: DraftingItem[]`; the `/api/handoffs`
  response carries it. `listBoardItems` already projects `#status`, so no
  projection change is needed.

### 3. Frontend rewire (`services.ts`, `store.ts`, `types.ts`)

- `fetchCandidates()` drops both `structuredClone(PIPELINE.discovery)` and
  `structuredClone(PIPELINE.drafting)`. Discovery comes from a new
  `fetchDiscovery()` calling `GET /api/discovery`; Drafting maps from the
  extended `/api/handoffs` `drafting` array.
- Store initial state (`store.ts:128`) seeds **empty** columns
  (`{ discovery: [], drafting: [], readyForDev: [], inReview: [] }`) instead of
  `structuredClone(PIPELINE)`, until the fetches resolve.
- Frontend `DiscoveryCard` / `DraftingCard` types updated to the real shapes.

### 4. Cleanup

- Delete `PIPELINE`, `HANDOFFS`, `REVIEW_BLOCKS` from `data.ts`. Keep
  `ARTICLE_HTML`, `TECH_EDITOR_NOTES`, `EMPTY_DEPLOY`.
- Remove now-unused `PipelineData`/handoff-doc imports left dangling.
- **Delete stale ledger rows** from the deployed `PublicationLifecycle` table:
  the `pewdiepie-archdaemon/odysseus` record and any other non-genuine test
  rows. One-off step via AWS CLI `delete-item` (or a tiny scripted scan +
  delete), run against the deployed table name. This is a data operation, not
  code — called out explicitly in the plan as a manual/confirmed step.

## Data flow (after)

```
Discovery   ── GET /api/discovery ──► searchEmergingRepos() ─► GitHub Search API
                                       └─ filter out all ledger repoNames
Drafting    ─┐
Ready-for-Dev─┤── GET /api/handoffs ──► listBoardItems() ─► PublicationLifecycle
In-Review   ─┘                          └─ bucketBoard() → {readyForDev,
                                                            drafting, inReview}
```

## Testing

- **Pure units (vitest, no AWS):**
  - `filterUnpublished` already tested; add a test for filtering against *all*
    ledger names if a new helper is introduced.
  - `bucketBoard` — extend to assert `DRAFTING` items land in `drafting`.
  - `buildMarkDraftingParams` — asserts the UpdateCommand shape (mirrors the
    existing `buildMarkPublishedParams` test).
  - `searchEmergingRepos` — mock Octokit, assert query + mapping to
    `RepoCandidate[]`.
- **Handlers:** `listDiscovery` handler — mock the search helper + ledger,
  assert 401 without JWT, assert filtered top-N payload. `draftArticle` —
  assert `markDrafting` is called at start.
- **Typecheck + full vitest suite green; production build OK** before merge.
- **Manual E2E (the original deferred verification):** trigger discovery on a
  real demo repo with a `DIFFPRESS.md`, resume the handoff, confirm the card
  moves Discovery-absent → Ready-for-Dev → Drafting → In-Review with a genuine
  Gemini headline (not the old "A Critical Read of…" template).

## Open implementation choices (for the plan, not blockers)

- Exact `listAllRepoNames` vs. reusing/scanning approach for the discovery
  dedupe filter — pick the lighter scan.
- Discovery result count (default 8) — tune in the plan.
